# AGENTS.md

本文件定义本项目后续设计、规划、开发和协作的固定规则。所有新增方案、代码、文档和资源都应遵循这些约束。

## Project Context

- 项目名称：Chrono Echo: Fractured Realms
- 项目类型：网页端 Roguelike 休闲游戏
- 当前方向：时间裂隙、时间回响、轻量构筑、短局循环、阶段式开发
- 目标体验：容易上手、单局节奏快、构筑有变化、局后有记忆，但不让玩家被复杂系统压垮

## Documentation First

- 所有核心设计必须落地到文档，不只停留在对话中。
- 新增或修改玩法、世界观、系统、数值、敌人、关卡、UI、技术方案时，应同步更新 `docs/` 下的对应文档。
- 文档应优先描述可执行内容，包括玩家行为、系统规则、输入输出、边界条件和开发阶段。
- 不采用过度抽象或只用于展示概念的设计描述。每个系统都应说明它如何进入实际游戏。

## Planning And Development Documents

- 规划和开发必须有执行文档支撑。
- 阶段开发前，应先维护开发计划，至少包含目标、范围、交付物、验收标准和暂不处理的内容。
- 推荐文档结构：
  - `docs/overview.md`：项目总览和核心方向
  - `docs/game-design.md`：玩法、系统、世界观、角色、敌人、关卡和构筑设计
  - `docs/development-plan.md`：阶段计划、任务拆分、验收标准
  - `docs/technical-plan.md`：技术架构、模块边界、数据结构、运行方式
  - `docs/changelog.md`：重要阶段变更记录
- 若文档结构后续调整，应先说明原因，并保持旧内容可追溯。

## Gameplay Design Rules

- 游戏是休闲 Roguelike，不应设计成高门槛、重策略、长学习曲线的复杂游戏。
- 每个系统都必须服务于玩家可感知的乐趣，避免只在设定上复杂但游玩中难以理解。
- 单局目标应保持清晰，推荐 10 到 15 分钟完成一局。
- 玩家在前 1 分钟内应能理解基本操作和当前目标。
- 构筑深度来自少量规则的组合，而不是大量难以记忆的系统。
- 时间主题应优先体现在玩家操作、战斗反馈和局后变化中，而不是只作为背景设定。
- 新机制默认采用最小可玩版本，确认体验有效后再扩展。

## Visual And UI Rules

- 禁止在项目文档、UI 文案、提交信息和代码注释中使用 emoji。
- 可以使用 SVG 绘制图标、示意图、流程图或简单视觉资产。
- Web 游戏界面应优先清晰、轻量、易读，避免装饰压过玩法。
- UI 控件应符合玩家直觉，核心操作必须明显且响应迅速。

## Git Workflow

- 项目必须使用 Git 管理。
- 每个阶段开发开始前，应明确阶段目标和验收标准。
- 当一个阶段达到可验证状态后，应进行 Git 提交。
- 提交应围绕完整阶段或清晰功能点，不提交无意义的零散修改。
- 不随意重写历史，不删除用户已有内容，除非得到明确确认。
- 提交信息应简洁描述阶段成果，例如：
  - `docs: add project collaboration rules`
  - `docs: define core gameplay design`
  - `feat: add combat prototype`

## Collaboration Rules

- 继续设计时，先把方案整理成可落地文档，再进入开发。
- 进入开发时，先确认阶段范围，避免一次性铺开过大。
- 如果出现复杂度过高的设计，应主动收敛到更轻量的版本。
- 对不确定内容，应先提出推荐方案，并说明取舍。
- 所有最终产出都应能被后续开发继续使用，而不是只用于讨论。
